# Explorer Tree: Design Decisions & Enhancement Ideas

## Current state

The Page Explorer is a Figma-like layers panel in the left sidebar that shows a **filtered component hierarchy** ŌĆö not a raw DOM tree. The goal: a designer sees `HeroSection > Card > Button`, not `<div> <div> <div> <div>`.

---

## Key design decisions

### 1. "Proof of intent" ŌĆö only show elements the developer wrote

The Babel transform (`next-plugin/loader.ts`) adds `data-source="file:line:col"` to every JSX element at compile time. A `<div>` with `data-source` was explicitly authored in user code. A `<div>` without it was auto-generated by a library or framework ŌĆö skip it.

This is the single most important filter. It eliminates the vast majority of noise.

### 2. Three types of host elements are shown

| Type | Condition | Example | Why |
|------|-----------|---------|-----|
| Design system components | Has `data-slot` attribute | `<div data-slot="card-title">` ŌåÆ **CardTitle** | Always relevant ŌĆö these are the editable pieces |
| Semantic HTML landmarks | Tag is `header`, `main`, `nav`, `section`, `article`, `footer`, `aside` | `<main>` ŌåÆ **\<main\>** | Structural context ŌĆö meaningful to designers, not noise |
| Author-written elements | Has `data-source` and tag isn't in skip list | `<div data-source="page.tsx:12:4">` ŌåÆ **\<div\>** | User deliberately wrote this element |

### 3. Everything else is hidden ŌĆö but children bubble up

When a fiber is filtered out (framework internal, anonymous wrapper, generic container), its children still walk and appear under the nearest visible ancestor. This means:

```
OuterLayoutRouter (hidden)
  InnerLayoutRouter (hidden)
    ClientPageRoot (hidden)
      <div> without data-source (hidden)
        HeroSection (visible) ŌåÉ bubbles up to root level
```

Nothing is "lost" ŌĆö components always appear, just without the noise wrapping them.

### 4. Framework internals are aggressively filtered

~50+ regex patterns suppress React and Next.js plumbing:
- Routing: `InnerLayoutRouter`, `OuterLayoutRouter`, `ClientPageRoot`, `SegmentViewNode`
- Boundaries: `Suspense`, `ErrorBoundary`, `LoadingBoundary`, `NotFoundBoundary`
- Wrappers: `Provider`, `Consumer`, `Fragment`, `Memo`, `ForwardRef`, `Lazy`
- Next.js: all `__next_` prefixed, `AppRouter`, `Router`, `Root`, `HotReload`

### 5. Single-child component collapsing

If a component wraps exactly one child component and has no text of its own, it's collapsed ŌĆö the child is promoted in its place. This prevents chains like `PageWrapper > ContentWrapper > InnerWrapper > ActualComponent` from bloating the tree.

### 6. Layout vs Page scope separation

The tree is split into **Layout** (collapsed by default) and **Page** (expanded) sections. Designers typically work on page content, not the persistent navigation shell. Layout auto-expands when the user selects something in it.

Scope is inferred from file paths: `layout.tsx` ŌåÆ layout scope, `page.tsx` ŌåÆ page scope. Instance source (where a component is *used*) beats definition source.

### 7. Visual hierarchy for designers

- **Design system components** (`data-slot`): bold weight, accent color, filled icon ŌĆö these are the editable primitives
- **User components**: normal weight, muted text, outline icon
- **HTML elements**: dimmed, box icon ŌĆö structural, less prominent
- **Text preview**: first ~40 chars shown inline for quick identification

### 8. Auto-expand and scroll to selection

When the user clicks an element in the viewport, the tree auto-expands the path from root to that node and scrolls it into view. This keeps tree and viewport in sync without manual navigation.

### 9. Framework-agnostic editor, framework-specific extraction

The tree extraction (fiber walking) lives in the framework plugin's `<Surface />` component (e.g. `next-plugin/surface.tsx`). The editor (`page-explorer.tsx`) only receives `ComponentTreeNode[]` over postMessage and is completely framework-agnostic. A future Vue or Svelte plugin would populate the same shape.

### 10. Fallback for non-React apps

When React fiber isn't available, a DOM-only fallback walks the tree showing only `data-slot` elements and semantic landmarks. Less rich but still functional.

## What's NOT shown today (and why)

| Excluded | Reason |
|----------|--------|
| Generic `<div>`, `<span>`, `<p>` without `data-source` | Auto-generated, noise |
| `<script>`, `<style>`, `<link>`, `<noscript>` | Document plumbing |
| `<html>`, `<body>`, `<head>`, `<br>`, `<hr>` | Document structure / void elements |
| Anonymous components (no name) | HOCs, arrow functions ŌĆö meaningless labels |
| Components without `data-instance-source` or `data-slot` | From node_modules or generated code |
| The `CodeSurface` component itself | Our own overlay |

## Enhancement ideas

### Enhancement 1: Filtered view / Full DOM tree toggle

**Problem**: The current tree only shows authored components and semantic landmarks. Sometimes a designer needs to see the actual DOM structure ŌĆö maybe to understand spacing issues, find a wrapper causing layout problems, or see what's between two visible nodes.

**Proposal**: Add a toggle in the explorer header ŌĆö something like an icon button that switches between "Component view" (current filtered tree) and "DOM view" (full tree). The DOM view would show all elements with `data-source`, including `<div>`, `<span>`, `<p>`, etc.

**Implementation approach**:
- The target app already walks the fiber tree. Add a mode flag to the `tool:requestComponentTree` message: `{ mode: "components" | "dom" }`.
- In "dom" mode, `processHostFiber` shows ALL elements with `data-source` (not just semantic tags and data-slot). This still filters out framework-generated containers (no `data-source`) but shows the full authored structure.
- The editor stores the mode in state and re-requests the tree on toggle.
- Visual difference: the "DOM view" uses a subtle background tint or border indicator so the user always knows which view they're in.
- The toggle should be a small icon in the explorer header, not a prominent switch ŌĆö most users will stay in component view.

**Open questions**:
- Should DOM view also show elements *without* `data-source`? That would be the truly full DOM ŌĆö but it's enormous and includes framework noise. Probably better to keep the `data-source` filter even in DOM mode, since that's "everything the developer wrote."
- Should the selection/hover still work the same way in DOM view? Yes ŌĆö same postMessage protocol, same `selectByTreeId`.

### Enhancement 2: Nested div collapsing ŌĆö `<div> +3`

**Problem**: Even in the filtered view, authored code often has deeply nested `<div>` chains for layout purposes:

```jsx
<div className="container mx-auto">
  <div className="flex gap-4">
    <div className="w-1/2">
      <div className="p-4 rounded-lg">
        <Card>...</Card>
      </div>
    </div>
  </div>
</div>
```

This appears in the tree as 4 levels of `<div>` before reaching `Card`. Each div matters for layout/styling, but showing them all as separate tree nodes creates visual noise and deep nesting.

**Proposal**: Collapse consecutive single-child `<div>` chains (and similar block-level elements) into a summary node: `<div> +3` meaning "div with 3 more nested divs inside before the next interesting content."

**Detection rules for collapsible chains**:
1. A chain of elements where each has exactly **one element child** (may also have text nodes)
2. All elements in the chain are **block-level layout elements**: `div`, `section`, `article`, `aside`, `main`, `header`, `footer`, `nav`, `form`, `fieldset`, `figure`, `details`
3. The chain has **2 or more** consecutive elements (a single div isn't worth collapsing)

**Display**:
```
Ō¢Č <div> +3          ŌåÉ collapsed (default for chains Ōēź3 deep)
  Card
  Button
```
Expanding shows the full chain:
```
Ō¢╝ <div> +3
  <div> container mx-auto
    <div> flex gap-4
      <div> w-1/2
        <div> p-4 rounded-lg
  Card
  Button
```

**The "+N" count** = number of additional nested elements in the chain (so `<div> +3` means 4 divs total, the visible one plus 3 more).

**Important**: Every div in the chain is still individually selectable/hoverable when expanded. The styles on each wrapper div are editable. The collapsing is purely a visual simplification in the tree ŌĆö it doesn't hide data.

**What triggers expansion**:
- Clicking the `+3` badge or the expand arrow
- Selecting any element inside the chain (via viewport click) ŌĆö auto-expands the chain to show the selected node
- The search/filter matching a class name or text inside the chain

**Implementation approach**:

*Option A ŌĆö Collapse in the target app (tree building)*:
Add a new node type or property to `ComponentTreeNode`:

```typescript
interface ComponentTreeNode {
  // ... existing fields ...
  /** If this is a collapsed chain, the intermediate nodes */
  collapsedChain?: ComponentTreeNode[];
}
```

The fiber walker detects single-child block-element chains and bundles them. The editor renders the collapsed view with expand/collapse.

*Option B ŌĆö Collapse in the editor (UI only)*:
The target app sends the full tree as-is. The editor's `PageExplorer` detects chains of single-child block elements and renders them collapsed. This keeps the protocol simple and the target app dumb.

**Recommendation**: Option B (editor-side). The protocol stays clean, the full tree is always available for DOM mode, and the collapsing logic is a rendering concern.

**What about styling information?**

The concern: if we collapse divs, how does the user know which ones have meaningful styles? Two approaches:

1. **Show class hints**: Display first few Tailwind classes or a style indicator on the collapsed summary: `<div> +3 ŌĆó flex gap-4 ŌåÆ p-4 rounded-lg`
2. **Style indicator dot**: A small colored dot next to divs that have non-default computed styles (display is not block, has padding, margin, background, border, etc.)
3. **Skip truly empty wrappers**: If a div has no classes, no inline styles, and no computed styles that differ from defaults, it could be skipped entirely (not even counted in the +N). But this is risky ŌĆö most authored divs do affect layout in some way.

Approach 1 (class hints) is probably the most useful for designers ŌĆö they can see at a glance what each wrapper does.

### Enhancement 3: Smart collapsing for other patterns

Beyond consecutive divs, other patterns could benefit from similar treatment:

**List items**: `<ul>` with 20 `<li>` children ŌĆö could show `<ul> (20 items)` collapsed, expanding to show items.

**Table rows**: `<tbody>` with many `<tr>` ŌĆö `<tbody> (15 rows)` collapsed.

**Repeated components**: If a parent has 10+ children of the same component type (e.g., 10 `NavLink` components), show `NavLink ├Ś10` collapsed.

These are less critical than the div chain problem but worth considering as the tree gets more sophisticated.

## Summary of proposed protocol/data changes

### ComponentTreeNode additions (for Enhancement 2, if Option A)

```typescript
interface ComponentTreeNode {
  // ... existing ...
  /** Intermediate nodes in a collapsed element chain */
  collapsedChain?: ComponentTreeNode[];
}
```

### New message type (for Enhancement 1)

```typescript
// Editor -> Target
| { type: "tool:requestComponentTree"; mode?: "components" | "dom" }
```

### No changes needed for Enhancement 2 with Option B

The editor handles collapsing purely in rendering ŌĆö no protocol changes.
